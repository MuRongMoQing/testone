# 使用版本化 SQL 迁移演进数据库

数据库结构通过独立、顺序编号的 SQL 文件和 `schema_migrations` 表演进，服务启动时只继续尚未完成的步骤。每个版本拆成预检、单条原子 DDL、事务化 DML 回填、切换和后续清理等可重入步骤，并记录步骤的开始、成功或失败状态；只有所有必需步骤明确成功后才把版本标记完成并开放业务流量。

MySQL 的许多 DDL 会隐式提交，因此系统不承诺包含多条 DDL 的整个版本可以像普通 DML 事务一样回滚。任一步骤失败时迁移器拒绝业务启动并如实报告已执行、未执行和无法自动回滚的内容；恢复通过修复后的可重入执行或生产前验证过的备份完成，不能把“拒绝启动”表述为“数据库已经完整回滚”。

生产迁移前必须验证备份和恢复步骤。现有 `users` 与 `goods` 通过专门迁移在新增结构中保守转换，不直接覆盖或删除；破坏性清理只能在切换、对账和独立用户审批后执行。C++ 代码不再拼接完整建表结构。
