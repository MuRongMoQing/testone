# 库存申请使用公共头和类型明细表

库存操作工作流采用以下结构：

- `stock_operation_requests` 保存申请类型、申请人、状态、审批超时、执行期限、复核链和通用审计关联。
- `inbound_request_details` 保存 SKU 或全新货物提案、数量、单位进价、币种和目标货位。
- `outbound_request_details` 保存 SKU、数量、单位售价和币种；仓库员工可提交的建议来源货位使用独立关联行保存，不构成最终分配。
- `transfer_request_details` 保存 SKU、数量、来源货位和目标货位。
- 审批决定、带序号的复核尝试、附件、库存预留和执行结果分别保存在独立关联表中，不覆盖申请或历史决定。

申请头及后续审批、预留、执行、流水、批次和余额均明确保存 `warehouse_id`；业务明细通过货位标识关联该仓库中的叶级位置。服务接口必须接收仓库上下文并校验仓库与货位归属，不能依赖隐含的“当前唯一仓库”。

出库、销售取出和移位的最终来源货位及数量由管理员或经理在批准时写入 `inventory_reservations`，而不是回写申请人提交的类型明细。代理商不能提交或读取建议来源货位和预留明细；员工建议不约束审批人，实际执行只能消耗仍然有效的批准预留。

首版规定每份申请只有一种操作类型且只针对一个 SKU，以使审批责任范围、库存预留、执行结果、复核次数和会计分录可以独立处理。包含多个 SKU 的同一业务单据拆分为多份申请；已通过安全审查的同一附件版本可经关联表连接这些申请，无须重复保存二进制内容。

申请与批准本身不写入已经发生的库存事实。只有实际执行成功后，系统才在同一事务中写入库存流水、批次建立或消耗分配、库存余额投影及适用的会计分录。该结构共享一致工作流，同时避免完全分表造成逻辑重复或单一巨表产生大量无约束空字段。

一份获批申请可以在执行期限内产生多条不可变执行结果。每次执行数量必须大于零，累计执行数量不得超过批准数量；每次执行分别、原子地写入对应库存与会计事实。累计数量不足时申请处于部分执行状态，出库或移位的剩余数量继续占用原预留；期限届满后取消未执行余量并释放其预留，余量如仍需执行必须重新申请。

最终审批前，申请人可以直接撤回自己的申请并保留完整历史。批准后申请内容不可修改或直接撤回；申请人只能另行提交剩余量取消申请，并按原申请的管理员优先、经理超时接管流程审批。取消获批只关闭尚未执行余量并释放对应预留，已经执行的结果保持不变；执行期限届满则无需额外审批自动取消余量。取消请求及决定使用独立关联记录。

每个审批决定和每次执行都必须携带唯一幂等键。服务端锁定申请及相关预留、余额和批次后重新校验当前状态与剩余数量；同一事务统一保存命令结果、业务事实和审计事件。重复提交同一键与相同内容时返回首次结果，不再次审批或变更库存；同一键对应不同内容时拒绝处理。管理员与经理并发作出最终审批时，以数据库中首先成功提交且满足约束的决定为准。

被拒申请修改后不得覆盖原行，而是创建新的申请版本。每个版本以 `root_request_id` 归属同一复核链，并以 `previous_request_id` 指向直接前一版本；原版本的业务字段、附件和审批结果保持不可变。新版本重新进入正常审批流程，连续复核拒绝次数按根申请链计算，第三次复核被拒后关闭该链的复核入口。
