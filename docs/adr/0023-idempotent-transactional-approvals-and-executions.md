# 审批与库存执行采用幂等事务命令

审批决定和每次库存执行必须携带客户端生成的唯一幂等键，并保存动作类型、操作者、规范化请求内容的哈希及首次处理结果。相同幂等键与相同内容的重试直接返回首次结果；相同键携带不同内容时返回冲突，不能复用该键执行另一动作。

服务端在单个 MySQL 事务中锁定申请、预留、库存余额及相关批次，重新校验权限、申请状态、执行期限、剩余批准数量和可用库存。校验通过后，事务统一写入审批或执行结果、库存流水、批次建立或消耗分配、余额投影、适用的会计分录、幂等结果及审计事件；任一写入失败则全部回滚。审批竞争由唯一最终决定约束裁决，执行竞争由剩余数量和预留约束裁决，只有首先成功提交且仍满足全部条件的事务生效。

该设计防止网络重试、重复点击和并发请求造成重复审批、重复入库、超量取出或库存与账册分离；代价是业务命令必须携带稳定标识，服务端还需长期保存足以重放响应的结果摘要。
