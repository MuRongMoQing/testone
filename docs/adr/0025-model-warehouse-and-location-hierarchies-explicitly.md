# 显式建模仓库组织树与仓内位置树

系统分别建立 `warehouses` 与 `storage_locations` 主表。仓库通过 `parent_warehouse_id` 形成总仓、区域仓和分仓等组织树；位置通过 `warehouse_id` 和 `parent_location_id` 形成库区、通道、货架及叶级货位等物理树。两种树承担不同含义，禁止把分仓当作货位或把货架当作仓库。系统拒绝父子循环及跨仓位置父子关系。

仓库和货位均使用永久唯一且不可修改的编码；显示名称和说明可以审计式修改。库存批次、余额、申请、预留、执行结果和库存流水均明确保存或可无歧义追溯 `warehouse_id` 与叶级 `location_id`，余额键至少包含仓库、叶级货位和 SKU。库存只能落在可用叶节点；存在库存、有效预留或已批准待执行移入时不能停用相关货位或仓库。停用只阻止新业务，不删除历史。

初始目标部署只创建一个根仓库，但首版目标数据库、领域服务和 HTTP API 将完整支持多个有效仓库，并要求显式仓库上下文，例如以 `/warehouses/{warehouse_id}` 作为库存和位置资源边界，同时提供仓库树与位置树的管理和查询接口。经理负责创建、调整和停用仓库及位置节点，所有变更进入追加式审计并提醒监察员。当前遗留实现尚不具备这些能力。

不同仓库之间的移动使用独立的跨仓调拨接口和状态模型，区分申请、双边审批、发运、在途、收货和差异处理，并由发出仓与接收仓分别确认。普通库内移位仅适用于同一仓库内的叶级货位，不得借它让目的仓库存提前增加。存在至少两个有效仓库且相关管理员、员工、货位和货物职责分配完成后，前端自动显示跨仓入口；否则只隐藏入口，后端能力和受控 API 仍存在。
