# 跨仓调拨采用双边审批和在途状态

跨仓调拨使用独立于普通库存申请与库内移位的聚合、接口和状态机，但可复用附件、安全扫描、幂等命令、审计及通知基础设施。一份调拨仍只针对一个 SKU，明确保存发出仓、接收仓、请求数量和双方责任范围；发出仓与接收仓不得相同。

该聚合、后端 API、数据库约束及集成测试均属于首版交付，不是仅供未来使用的占位接口。初始部署只有一个根仓库时前端隐藏调拨入口；经理建立第二个有效仓库并完成相关管理员、员工、货位和货物职责分配后，入口自动可用。后端始终按权限和组织完整性校验受控 API 请求，不依赖前端隐藏实现安全。

只有发出仓内负责该 SKU 的仓库员工可以创建调拨申请。管理员、经理、会计、监察员和代理商均不能成为申请人；管理员或经理提出业务需求时，先向负责员工发送留痕通知，后续申请关联该通知。该限制避免管理员既发起又审批自己的跨仓业务，也让发出仓的实际保管责任人确认请求内容。

发出仓管理员先批准调拨数量并在来源货位建立预留，接收仓管理员再批准接收并指定目标叶级货位。两侧审批必须由不同人员完成；如果同一管理员的仓库子树同时覆盖两端，则经理承担接收侧第二审批。只有两侧均批准后，发出仓员工才能实际发运。发出审批、接收审批及其并发重试均遵守幂等事务规则。

发出仓管理员暂停时，发出侧审批直接转交经理，接收侧仍由接收仓管理员完成；若接收仓管理员也不可审批，则调拨保持待审批，不能用同一经理同时替代两侧。该规则优先保证双边审批由不同人员完成。

每次分批发运原子地消耗来源预留、减少发出仓余额，并将数量及对应先进先出批次成本转入在途库存；此时不得增加接收仓余额。接收仓员工可以分批收货，每次收货附独立业务单据，并将确认数量从在途库存转入指定目标货位的接收批次。发运和收货分别保存不可变执行结果、库存流水、批次关联与审计事件。

每次发运同时借记在途库存并贷记发出仓库存资产；每次收货同时借记接收仓库存资产并贷记在途库存。调拨沿用发出批次已经锁定的原币成本、汇率和人民币成本，不产生新汇率报价、收入、销售成本或利润。库存、在途数量和分录必须在各自执行事务中保持一致。

累计实收数量与对应在途数量不一致时建立调拨差异，保留期望量、实收量、相关执行和单据，不自动调整、删除或认定损耗。发出与接收两侧负责人分别提交说明和证据；短少数量在解决前继续保持在途，超过对应在途数量的实物进入不可用且不入账的调拨隔离库存。

延迟到货可以继续按原调拨收货；实物退回发出仓须经双边确认并追加反向调拨流水；确认损坏或丢失须经对应会计组三人中至少两人同意、经理批准并提醒监察员，再追加库存与会计调整。来源识别错误或超收实物必须关联正确调拨，或另行通过正常入库审批。所有解决方式均保留原发运、收货、差异、流水和分录。该设计增加双边调查和隔离状态，但防止未查明差额被静默转为损失或可售库存。
