# 采用领域模块化单体与端口适配器

系统继续部署为一个 C++17 服务进程和一个 MySQL 数据库，但将当前集中在 `src/main.cpp` 的套接字、HTTP、鉴权、SQL、迁移和业务逻辑拆入领域、应用、API、基础设施与启动组合根。`main.cpp` 只负责调用组合根和控制进程生命周期；领域层不包含 MySQL、HTTP、供应商 SDK 或全局连接细节。

领域层按身份权限、货物目录、仓库层级、申请审批、库存、跨仓调拨、会计、附件及审计通知等业务能力组织。应用层编排命令和查询并拥有工作单元边界；API 层只处理协议与 DTO；基础设施层实现 MySQL、ClamAV、汇率、通知和账册导出端口。依赖只指向领域核心，CMake 以独立库目标和最小链接关系执行该约束。

申请、预留、批次、流水、余额、分录、审计和事务性发件箱的原子写入由应用工作单元统一控制。仓储和外部适配器不得自行提交业务事务，也不得越过应用服务直接组合跨模块写入。首版不拆微服务，以避免库存和会计强一致业务引入分布式事务；未来只有在模块边界稳定且确有独立伸缩或部署需求时，才评估进程拆分。

重构采用可验证的纵向切片而非一次性重写，先抽离配置、组合根和 MySQL 工作单元，再迁移认证、查询、申请执行及其他业务模块。该选择增加接口和构建目标数量，但让复杂权限、库存与会计规则可以脱离网络和数据库进行测试，并降低继续扩展单一 `main.cpp` 的变更风险。
